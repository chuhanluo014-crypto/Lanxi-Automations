# 候选选题池使用与生成规则

## 文件

- `topic_candidates_v1.csv`：候选选题池，已按 `LanXi/计划/蓝汐-30天故事线表.md` 的 Day 顺序排序，并补充故事线字段。

## 为什么要按故事线排序

实际发布时，蓝汐账号需要先建立人设，再建立互动，再沉淀栏目，最后进入复盘和下一阶段入口。因此生产顺序应优先看 `故事线顺序` 和 `故事线_day`，而不是优先看栏目。

推荐使用顺序：

1. 先用 `topic_candidates_v1.csv` 做发布排期。
2. 再用 `栏目` 字段检查内容配比，避免某一类内容过多。
3. 最后用 `优先级` 和 `状态` 筛选进入内容生成的选题。

## 故事线字段来源

以下字段来自 `LanXi/计划/蓝汐-30天故事线表.md`：

- `故事线顺序`
- `故事线节点`
- `故事线推荐内容形式`
- `故事线原文案核心`
- `故事线原互动钩子`
- `故事线栏目雏形`

候选选题不是覆盖 D1-D30 的完整日历，而是从故事线中抽取适合进入 Week 6/W7 内容生产的候选点。因此当前没有覆盖 D1-D3、D13-D15、D28 等所有 Day。后续如果要做完整 30 天发布日历，需要在这些空缺 Day 上继续补选题。

## 参考样本_id 的来源

`参考样本_id` 不是直接复制样本内容，也不是数据库自动生成的精确公式。它是从 `Database/Datapool/features/xhs_text_features_v1.csv` 中按以下规则筛出来的参考样本编号：

1. 先过滤样本：
   - `engagement_level = 高互动`
   - `keep_status = 保留`
   - `risk_level = 低`
2. 再匹配选题需要的结构特征：
   - `title_template` 是否接近当前选题的标题模板
   - `content_structure` 是否接近当前选题的内容结构
   - `scene` 是否接近当前选题的使用场景
   - `pain_point` 是否接近当前选题要承接的用户情绪/需求
3. 最后人工选择 1-3 个最有参考价值的样本 ID，写入 `参考样本_id`。

这些 ID 的作用是追溯“这个选题借鉴了什么样的爆款结构”，不是让后续内容复刻原样本标题、正文、画面或 IP 外观。

## 后续更严谨的自动评分规则

如果后续要把 `参考样本_id` 自动化，可以用以下评分规则：

```text
基础过滤：
engagement_level = 高互动
keep_status = 保留
risk_level = 低

匹配得分：
content_structure 匹配：+3
title_template 匹配：+2
scene 匹配：+2
pain_point 匹配：+2
hook 匹配：+1
engagement_score 排名前 20%：+1

每条候选选题取总分最高的 2-3 个 sample_id。
```

当前 v1 选题池尚未使用脚本自动评分，属于“数据筛选 + 人工结构匹配”的第一版。
